# ADR-0016: 익명 반응("불 지피기")을 좁은 RPC로 받는다

- 상태: 수락됨
- 날짜: 2026-06-19

## 맥락 (Context)

"불 지피기"는 **익명 방문자가 누르는 반응**이다. 카운트를 영속하려면 익명이 DB에 써야 한다. 그런데 우리 보안 모델은 [ADR-0013]에 따라 *"읽기는 누구나 published, 쓰기/수정/삭제는 인증된 나만"* 이다(RLS). 익명에게 `posts` 테이블 쓰기를 그냥 열면 카운트뿐 아니라 **글 수정·삭제까지** 노출된다 — 받아들일 수 없다.

## 결정 (Decision)

- 테이블 직접 쓰기는 **계속 막는다**(RLS 그대로).
- 대신 **`SECURITY DEFINER` 함수 하나(`increment_stokes(slug)`)** 만 익명(`anon`)에 노출한다. 이 함수가 하는 일은 **"발행된 글의 `stokes` +1"** 딱 하나다.
- 클라이언트는 Server Action(`stokePost`)을 통해 이 함수만 호출한다.

## 이유 (Rationale)

- **최소 권한**: 익명은 테이블을 못 만지고, 미리 정해둔 안전한 동작 하나만 실행한다. 기존 RLS 모델이 깨지지 않는다.
- 함수는 소유자 권한으로 돌아 RLS를 우회하지만, **하는 일이 고정**이라 우회 가능한 표면이 없다. `search_path`를 `public`으로 고정해 함수 하이재킹도 막는다.
- 기본(PUBLIC) 실행권은 거두고 `anon`·`authenticated`에만 명시 부여해, 권한을 분명히 한다.

## 중복 처리 (Trade-off)

- 중복은 **브라우저당 1회**(localStorage)로 가볍게 막는다. 우회하면 더 누를 수 있지만, 개인 블로그에서 카운트가 약간 부풀려지는 정도라 **수용**한다 — 정밀 집계가 목적이 아니라 "온기를 전하는" 가벼운 신호다.
- 진짜 1인 1회·통계가 필요해지면, 클릭마다 행을 쌓는 식별자별 `post_stokes` insert 테이블로 후속에서 옮길 수 있다.

## 고려한 대안 (Alternatives)

- **익명에게 테이블 UPDATE 허용** — 가장 단순하지만 글 전체에 대한 쓰기를 여는 셈이라 과한 권한. 기각.
- **클릭마다 행을 쌓는 `post_stokes` insert 테이블** — 식별자별 중복 방지·통계엔 유리하나, 지금 필요(가벼운 카운트)에는 과하다. 여지로만 남긴다.

## 결과 (Consequences)

- 마이그레이션 `0004_stokes.sql`: `posts.stokes` 컬럼 + `increment_stokes` 함수 + 권한.
- 쓰기: Server Action `stokePost(slug)` → RPC. 읽기: `queries`가 `stokes`를 함께 가져온다.
- `StokeButton`이 카운트를 영속·복원한다(브라우저당 1회, 카운트는 지핀 사람에게만 표시).
