# ADR-0017: 에디터 이미지는 Storage에 올리고 본문엔 URL만 둔다

- 상태: 수락됨
- 날짜: 2026-06-20

## 맥락 (Context)

에디터에서 이미지를 바로 넣고 싶다(드래그드랍·붙여넣기·`/이미지`). 본문은 마크다운으로 저장한다([ADR-0014](0014-markdown-content.md)). 그럼 이미지를 **어디에, 어떤 형태로** 둘지가 남는다.

## 결정 (Decision)

- 이미지 **실물은 Supabase Storage 공개 버킷(`post-images`)** 에 저장하고, 본문엔 **공개 URL만** `![](url)` 표준 마크다운으로 둔다.
- **업로드/수정/삭제는 인증 사용자(나)만**(Storage 정책), **읽기는 공개**.
- 업로드 시 **이미지 종류·크기(10MB) 가드**를 둬 비이미지·초대용량을 막는다.
- 큰 이미지는 올리기 전에 **긴 변 1600px로 자동 축소**해 용량·로딩을 최적화한다(gif·svg는 원본 유지).

## 이유 (Rationale)

- 본문이 마크다운이라 이미지도 **표준 링크**로 남으면 읽기 경로(공개 페이지)가 그대로 동작하고 **이식성**이 유지된다(에디터·DB와 무관하게 이미지는 URL).
- 큰 바이너리는 DB가 아니라 **Storage가 맞는 자리**다.
- 쓰기를 인증으로 좁히는 건 우리 보안 철학과 같다 — 읽기는 열고 쓰기는 좁게([ADR-0013], [ADR-0016](0016-stoke-anon-write.md)).

## 트레이드오프 (Trade-off)

- **고아 이미지**: 업로드는 떨구는 즉시 일어나므로, 본문에서 지우거나 글을 안 쓰면 파일이 참조 없이 남는다. 인라인 GC는 새 글(아직 id 없음)·버려진 드래프트를 못 잡아 지저분하다. 그래서 **"본문에서 참조 안 되는 파일을 찾아 지우는 스윕"을 별도 후속**으로 둔다(읽기·삭제만이라 비용도 사실상 0).
- **레이아웃**: 마크다운 저장이라 노션식 **이미지별 정렬·리사이즈는 포기**한다. 대신 공개 화면에서 **일관된 기본 스타일**(가운데·본문폭 등)로 보여, 콘텐츠 블로그 톤에 맞춘다.

## 고려한 대안 (Alternatives)

- **이미지를 본문에 base64로 인라인** — 글이 비대해지고 캐싱·성능에 나쁨. 기각.
- **이미지별 정렬/너비를 HTML이나 커스텀 문법으로 저장** — 마크다운 이식성·단순함이 깨지고 독자 측 살균 부담이 커서 개인 블로그엔 과함. 기각.

## 결과 (Consequences)

- 마이그레이션 `0005_storage.sql`: 공개 버킷 + 정책(쓰기는 인증, 읽기는 공개).
- `features/posts/upload.ts`: 업로드 헬퍼(종류·크기 가드 + 큰 이미지 자동 축소). `MarkdownEditor`의 Crepe `onUpload`에 연결.
- 공개 화면 `.prose img`: `max-width:100%`·가운데 — 작은 건 작게, 큰 건 본문폭에 맞춰 일관 표시.
- 후속: **미사용 이미지 스윕**(별도 이슈).
