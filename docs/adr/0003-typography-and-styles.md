# ADR-0003: 타이포그래피와 스타일 구조

- 상태: 수락됨
- 날짜: 2026-06-17

## 맥락 (Context)

글이 주인공인 블로그다. 그런데 처음엔 제목·본문·코드·UI에 폰트를 제각각 쓰다 보니, 폰트가 여러 개로 늘면서 위계가 흐려지고 화면이 산만했다. 내가 보기 좋다고 느낀 개인 블로그들은 폰트를 적게, 역할을 분명히 쓴다 — 본문은 차분한 한 벌, 제목·식별 요소는 개성 있는 한 벌. 그 방향이 맞다고 봤다.

[ADR-0001](0001-brand-and-logo.md)에서 정한 Frost 팔레트와 2계층 토큰 위에, 실제 타이포그래피와 스타일 파일 구조를 확정한다.

## 결정 (Decision)

- **폰트는 역할당 한 벌, 총 2역할로 압축**
  - 본문: **Pretendard** — 한글·라틴이 균질하고 긴 글에서 피로가 적다.
  - 제목·날짜·UI 라벨: **Gmarket Sans** — 기하학적이고 친근한 디스플레이체. 본문과 대비를 줘 위계를 만든다.
  - 코드: 시스템 모노스페이스 — 코드 비중이 낮아 별도 웹폰트를 로드하지 않는다.
- **폰트는 `next/font`로 셀프 호스팅** — CDN 링크 대신 같은 출처에서 제공한다.
- **본문은 `.prose`로 가볍게 자체 정의** — 큰 줄높이(1.95)와 명확한 위계.
- **스타일 파일 분리** — `tokens.css`(토큰), `animations.css`(인터랙션), `globals.css`(기본·본문)로 관심사를 나눈다.
- **테마 초기화** — 저장된 테마(없으면 OS 설정)를 하이드레이션 전 동기 스크립트로 적용한다.

## 이유 (Rationale)

- 폰트를 줄이고 역할을 분명히 하면 위계가 또렷해지고, 찍어낸 듯한 인상 대신 의도가 보인다. 잘 만든 서비스의 결을 따른다.
- 본문과 제목을 다른 폰트로 두면 한 벌일 때보다 화면에 표정이 생긴다.
- 코드용 폰트를 시스템 모노로 두면 추가 네트워크 비용 없이 어디서나 자연스럽다.
- CDN으로 폰트를 받던 방식의 "공유 캐시" 이점은 브라우저가 캐시를 사이트별로 분리하면서 사라졌다([Chrome](https://developer.chrome.com/blog/http-cache-partitioning)). 외부 출처는 글자 렌더 전 추가 연결과 장애 위험이 있어, 같은 출처에서 받고 `next/font`가 preload·size-adjust 폴백으로 레이아웃 흔들림을 줄이게 했다.
- 토큰/애니메이션/기본을 파일로 나누면 `globals.css`가 비대해지지 않고, 고칠 곳을 빨리 찾는다.
- 테마를 하이드레이션 전에 칠하면 새로고침 시 흰 화면 깜빡임(FOUC)이 사라진다.

## 고려한 대안 (Alternatives)

- **단일 폰트로 통일** — 가장 단순하지만 제목·본문의 위계와 개성이 약해 기각.
- **역할별로 폰트를 더 늘리기(코드·숫자 전용 등)** — 처음 상태. 화면이 산만하고 로드 비용만 커져 기각.
- **CDN(동적 서브셋) 로드** — 한글 용량은 줄지만 제3자 의존·추가 연결이 생기고, 캐시 공유 이점이 사라져 셀프 호스팅으로.

## 결과 (Consequences)

- 토큰: `src/styles/tokens.css` (폰트·색·`@theme` 매핑)
- 인터랙션: `src/styles/animations.css`
- 기본·본문: `src/app/globals.css`
- 폰트 등록·셀프 호스팅: `src/app/fonts.ts` (파일은 `src/fonts/`)
- 테마 초기화: `src/app/layout.tsx`
- 후속: 한글 폰트를 서브셋으로 줄여 용량 최적화.
