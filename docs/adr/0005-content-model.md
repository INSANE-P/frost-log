# ADR-0005: 콘텐츠 모델과 빈 상태

- 상태: 수락됨
- 날짜: 2026-06-17

## 맥락 (Context)

블로그는 두 흐름을 담는다 — 매일의 기록(journal)과 공들이는 이야기(post). [ADR-0002](0002-stack-and-structure.md)에서 읽기 경계와 폴더 구조를 정했고, 여기선 그 위에서 콘텐츠 데이터 모델과 빈 화면을 어떻게 다룰지 정한다.

## 결정 (Decision)

- **기록과 이야기는 하나의 `Entry` 타입을 공유하고 `type`으로 구분한다.** 상세·작성 레이아웃을 공유하기 위함(차이는 날짜 부각 정도).
- **대표 이미지는 본문 첫 이미지를 자동으로 끌어올린다(`coverOf`).** 글 안에 이미지를 넣기만 하면 목록에서 대표로 보인다.
- **읽기는 `queries.ts` 한 곳에서.** 지금은 mock, Supabase 연결 시 이 파일 내부만 교체한다.
- **빈 상태 문구는 방문자 시점으로 쓴다.** 관리자에게 지시하는 말("글을 등록하세요")이 아니라 방문자가 읽는 안내.

## 이유 (Rationale)

- 기록과 이야기는 성격만 다르지 글이라는 본질은 같다. 한 모델로 두면 상세·에디터를 두 번 만들지 않는다.
- 대표 이미지를 따로 지정하게 하면 글쓰기에 마찰이 생긴다. 본문에 넣은 첫 이미지를 자동으로 쓰면 손이 덜 간다.
- 읽기를 한 파일로 모으면 나중에 DB로 바꿀 때 화면 코드는 건드리지 않는다.
- 빈 화면도 결국 방문자가 본다. 운영자에게 하는 말이 노출되면 읽는 사람이 헷갈린다.

## 고려한 대안 (Alternatives)

- **기록·이야기를 별도 타입/테이블로 분리** — 초기엔 명확하지만 레이아웃·쿼리가 두 배가 돼 기각.
- **대표 이미지를 항상 수동 지정** — 정확하지만 글쓰기 마찰이 커 기각.

## 결과 (Consequences)

- 타입: `src/features/posts/types.ts` (`Entry`, `coverOf`, `entryHref`)
- 읽기 경계: `src/features/posts/queries.ts` (mock: `mock.ts`)
- 공용 UI: `src/components/ui/EmptyState.tsx`, `SectionLabel.tsx`
- 후속: 본문은 임시 HTML → Supabase 연결 시 Tiptap JSON 렌더로 교체.
